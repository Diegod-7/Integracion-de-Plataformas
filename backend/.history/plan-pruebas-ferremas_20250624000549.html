1. Introducción

1.1 Resumen ejecutivo

Este documento presenta el Plan de Pruebas de Software para el Sistema de E-commerce Ferremas, una plataforma completa de comercio electrónico desarrollada con Node.js, Express.js, PostgreSQL y integración con Transbank para pagos.

El plan establece una estrategia integral de testing que abarca 23 casos de prueba principales con más de 500 validaciones individuales, cubriendo aspectos funcionales, de seguridad, rendimiento e integración. Este es un plan maestro que define el alcance completo del esfuerzo de pruebas para garantizar la calidad y confiabilidad del sistema antes de su despliegue en producción.

1.2 Alcance de las pruebas

1.2.1 Elementos de pruebas

Los siguientes módulos y componentes serán sometidos a pruebas:

  • Módulo de Autenticación: Registro, login, validación JWT, seguridad de sesiones
  • Módulo de Productos: CRUD, búsqueda, filtros, paginación, validaciones
  • Módulo de Newsletter: Suscripciones, validación de emails, gestión de listas
  • Módulo de Pagos (Webpay): Transacciones, confirmaciones, integración Transbank
  • API REST: Endpoints, validaciones, respuestas HTTP, manejo de errores
  • Base de Datos: Integridad, consistencia, performance de consultas
  • Seguridad: Inyección SQL, XSS, autenticación, autorización

1.2.2 Pruebas funcionales

Las pruebas funcionales utilizan técnicas de "Caja Negra" para verificar el comportamiento del sistema sin inspeccionar detalles internos. Se utilizan los siguientes tipos de datos de entrada:

  • Valores normales: Datos típicos de uso cotidiano
  • Valores límites: Valores en los bordes de rangos aceptables
  • Valores de borde: Valores justo dentro y fuera de límites
  • Valores ilegales: Datos inválidos para probar manejo de errores

3. Estrategia de Pruebas

La estrategia de pruebas se basa en un enfoque integral que combina pruebas automatizadas con Jest y Supertest, cubriendo múltiples dimensiones de calidad del software.

3.1 Tipos de pruebas

3.1.1 Pruebas funcionales

Aspecto Descripción
Objetivo de la prueba: Asegurar la funcionalidad completa del sistema incluyendo navegación, ingreso de datos, procesamiento y recuperación de resultados. Validar que las reglas de negocio se implementen correctamente.
Técnica a utilizar: Ejecutar cada caso de uso usando datos válidos e inválidos para comprobar resultados esperados y manejo de errores. Utilizar Jest y Supertest para automatización de pruebas de API.
Criterio de validación: Todas las pruebas planificadas ejecutadas correctamente. Todos los defectos identificados y documentados. Cobertura de código superior al 80%.
Consideraciones especiales: Validación especial para integración con Transbank. Pruebas específicas para diferentes formatos de email y validaciones de productos.

3.1.2 Pruebas de rendimiento (Performance)

Enfocadas en verificar tiempos de respuesta, throughput y comportamiento bajo carga. Se ejecutan pruebas de stress con hasta 50 usuarios concurrentes.

3.1.3 Pruebas de seguridad y de acceso a datos

Suite completa de pruebas de seguridad incluyendo inyección SQL, XSS, validación de autenticación y autorización.

6. Entregables

6.1 Plan de pruebas

Este documento constituye el plan maestro de pruebas que será distribuido a todos los stakeholders del proyecto.

6.1.1 Criterio de entrada para el "Plan de pruebas"

  • Ambiente de pruebas configurado y funcional
  • Base de datos de pruebas con datos representativos
  • Código fuente estable y deployado en ambiente de pruebas
  • Herramientas de testing instaladas y configuradas

6.1.2 Criterio de salida para el "Plan de pruebas"

  • Ejecución exitosa del 100% de casos de prueba planificados
  • Cobertura de código superior al 80%
  • Todos los defectos críticos y altos resueltos
  • Defectos medios y bajos documentados y priorizados
  • Pruebas de performance dentro de parámetros aceptables
  • Pruebas de seguridad sin vulnerabilidades críticas

6.1.3 Criterio de suspensión y resumisión

Suspensión: Se suspenderán las pruebas si se detectan más de 5 defectos críticos o si el ambiente de pruebas presenta inestabilidad mayor al 20% del tiempo.

Resumisión: Las pruebas se reanudarán cuando todos los defectos críticos sean corregidos y el ambiente de pruebas sea estable por al menos 24 horas continuas.

6.2 Resultados de las pruebas

Los resultados se registrarán y reportarán utilizando:

  • Reportes HTML generados por Jest
  • Reportes Excel con métricas detalladas
  • Dashboard de cobertura de código
  • Logs detallados de ejecución de pruebas

6.3 Reporte de defectos

Los defectos se registrarán con la siguiente información:

  • ID único del defecto
  • Severidad (Crítica, Alta, Media, Baja)
  • Descripción detallada
  • Pasos para reproducir
  • Resultado esperado vs resultado actual
  • Capturas de pantalla o logs relevantes

7. Anexos

7.1 A: Tareas del proyecto

Lista completa de tareas relacionadas con el Plan de pruebas:

  • Diseño e implementación de 23 casos de prueba principales
  • Desarrollo de más de 500 validaciones individuales
  • Configuración de ambiente de pruebas automatizadas
  • Implementación de pruebas de seguridad (SQL Injection, XSS)
  • Desarrollo de pruebas de performance y stress
  • Creación de generadores de reportes Excel automatizados
  • Configuración de Jest con cobertura de código
  • Implementación de secuenciador personalizado de pruebas

7.2 B: Pruebas de rendimiento (performance)

Aspecto Descripción
Objetivo de la prueba: Comprobar tiempos de respuesta, throughput y comportamiento bajo carga normal y de stress. Validar que el sistema mantenga performance aceptable con hasta 50 usuarios concurrentes.
Técnica a usar: Ejecutar pruebas automatizadas con Jest que simulen múltiples usuarios concurrentes. Medir tiempos de respuesta para cada endpoint. Monitorear uso de memoria y CPU.
Criterio de validación: Tiempo de respuesta promedio < 2 segundos. Degradación de performance < 20% bajo carga. Sin memory leaks detectados.
Consideraciones especiales: Pruebas deben ejecutarse en ambiente dedicado. Base de datos debe contener volumen representativo de datos de producción.

7.3 C: Pruebas de seguridad y de control de acceso

Aspecto Descripción
Objetivo de la prueba: Verificar resistencia a ataques de inyección SQL, XSS, y validar correcta implementación de autenticación y autorización. Asegurar que solo usuarios autorizados accedan a funcionalidades específicas.
Técnica a usar: Ejecutar suite de 100+ pruebas de seguridad incluyendo payloads maliciosos, validación de tokens JWT, pruebas de autorización por roles, y verificación de sanitización de inputs.
Criterio de validación: Cero vulnerabilidades críticas o altas. Correcta validación y sanitización de todos los inputs. Autenticación y autorización funcionando según especificaciones.
Consideraciones especiales: Pruebas deben incluir vectores de ataque actualizados según OWASP Top 10. Validación especial para integración con Transbank.
